# Phase 23, Plan 1: Multi-Agent Layer

## Goal
Bring draht's multi-agent capabilities from basic subagent spawning to a full orchestration layer, closing the gap with the Claude Code architecture diagram. The existing `subagent.ts` builtin handles process spawning; this phase adds the coordination, communication, isolation, and safety primitives around it.

## Context
- `packages/coding-agent/src/core/builtins/subagent.ts` already spawns isolated draht processes (single, parallel, chain modes)
- Shipped agents exist in `packages/coding-agent/agents/` (architect, implementer, reviewer, etc.)
- The `/agent` command and input interception are already wired
- Missing: structured coordination protocol, inter-agent messaging, task board, worktree isolation, permission gate
- Reference: Claude Code Architecture diagram (Input Layer â†’ Multi-Agent Layer â†’ Execution Layer)

## Architecture

### 1. FSM Protocol (`src/core/multi-agent/fsm.ts`)
State machine for agent lifecycle coordination.

States: `IDLE â†’ REQUEST â†’ WORKING â†’ WAIT â†’ RESPOND â†’ IDLE`

- `IDLE`: Agent is available for work
- `REQUEST`: Task has been assigned, agent is initializing
- `WORKING`: Agent is executing (maps to subagent process running)
- `WAIT`: Agent is blocked, waiting for input from another agent
- `RESPOND`: Agent has completed, result is ready for consumption

Each agent instance gets a state machine. The orchestrator observes transitions via the event bus.

### 2. Teammate Mailbox System (`src/core/multi-agent/mailbox.ts`)
Pub/sub message passing between agents within a session.

- In-process message queue (no Redis dependency for single-machine use)
- Each agent gets a named mailbox
- Messages are typed: `TaskRequest`, `TaskResult`, `DataExchange`, `Abort`
- Mailboxes are created on agent spawn, destroyed on agent exit
- Optional persistent backing via filesystem for crash recovery

### 3. Autonomous Task Board (`src/core/multi-agent/task-board.ts`)
Self-service task assignment with atomic locking.

- Tasks are posted to the board with requirements (agent type, capabilities)
- Agents self-assign tasks that match their capabilities
- Atomic lock prevents double-assignment (file-based lock for cross-process safety)
- Board state is observable via event bus
- Integrates with existing parallel mode in subagent.ts

### 4. Worktree Isolator (`src/core/multi-agent/worktree.ts`)
Git worktree management for conflict-free parallel work.

- Creates `git worktree add` for each agent task
- Agents work in isolated worktrees instead of the main working directory
- On completion: merge worktree back to source branch
- Conflict detection on merge â€” reports conflicts instead of force-resolving
- Cleanup: remove worktree after merge or on abort
- Falls back to shared worktree (current behavior) when git is unavailable

### 5. Permission Gate (`src/core/multi-agent/permission-gate.ts`)
Three-tier permission system for tool execution.

- `deny`: Tool call is blocked (e.g., `rm -rf /`)
- `allow`: Tool call proceeds without confirmation
- `approve`: Tool call requires user confirmation before execution

Configuration via `.draht/permissions.yml` (project) and `~/.draht/agent/permissions.yml` (global):
```yaml
rules:
  - tool: bash
    pattern: "rm -rf *"
    action: deny
  - tool: bash
    pattern: "git push *"
    action: approve
  - tool: read
    action: allow
  - tool: write
    paths: ["src/**"]
    action: allow
  - tool: write
    action: approve
```

Rules are evaluated top-to-bottom, first match wins. Default: `approve` for bash, `allow` for read/edit/write within project.

## Risks
- Git worktree operations can be slow on large repos â€” keep isolation opt-in
- File-based locking for task board may have race conditions under high concurrency â€” acceptable for initial implementation, can move to advisory locks later
- Permission gate YAML parsing adds a dependency â€” use existing picomatch already in the dependency tree
